# Team Territory — правила и бизнес-требования

Документ фиксирует задумку многопользовательской игры на общем поле **квадратной сетки** динамического размера **G×G** (см. **п. 4.1–4.2**) с командами разных цветов. Реализация (API, WebSocket, БД) выполняется отдельно по этому ТЗ.

---

## 1. Цель игры

За время матча команды «закрашивают» клетки поля своим цветом. Победитель определяется **по числу клеток** данного цвета на момент окончания матча. Игроки победившей команды получают награду в алмазах (см. п. 9).

---

## 2. Команды и цвета

| Команда | Цвет (логический) | Макс. команд в матче |
|--------|-------------------|----------------------|
| A | Красный | 4 |
| B | Синий | |
| C | Зелёный | |
| D | Фиолетовый | |

- В матче участвуют **от 2 до 4 команд** (задано настройкой лобби/режима).
- В каждой команде **≥ 1 игрок** (боты в v1 не обязательны).
- Распределение по командам: **серверное**, сбалансированное по числу игроков (округление: более «полные» команды не должны отличаться более чем на 1 игрока, если возможно).

---

## 3. Лобби и старт матча

1. Игроки заходят в комнату матча (или общий матчмейкинг — на усмотрение реализации).
2. **В продакшене** (флаг debug-лобби **выключен**, см. п. 3.1) **старт возможен**, если одновременно:
   - задействовано **не менее 2 команд** с хотя бы одним игроком в каждой;
   - **общее число игроков ≥ 2**;
   - (опционально, режим «качество»): **≥ 3 игрока** или **≥ 4 клеток на команду в среднем** — флаг режима в конфиге.
3. Каждый игрок нажимает **«Готов»**. Пока не готов — в матч не попадает (или помечается неактивным — одна политика на весь проект).
4. Когда **все принятые в матч игроки** нажали «Готов» (или сработал **таймаут готовности**, например 60 с, и стартуют только готовые) — сервер вычисляет **`G`** по **п. 4.1**, переводит комнату в фазу **«Идёт матч»** и рассылает начальное состояние поля (**включая `G` и `CELL_TOTAL`**).

**Таймаут лобби:** если за N минут не набран минимум стартовых условий — комната закрывается без награды.

### 3.1 Временный debug-режим лобби (один разработчик / тест)

**Статус:** **временное** решение только для этапа разработки и ручного тестирования. **Перед выкладкой в продакшен** режим **обязан быть отключён**, а правила п. 3 п.2 — **восстановлены** без обходов (или переменная окружения / фича-флаг **жёстко запрещены** в prod-сборке).

**Зачем:** при тестировании в одиночку нельзя удовлетворить требованию «≥ 2 игрока» и «≥ 2 команды», из-за чего невозможно зайти в матч.

**Поведение при включённом debug** (например `TEAM_TERRITORY_DEBUG_SOLO_LOBBY=1` **только** в `.env` разработки / docker-compose override для dev):

- Допускается **старт матча с одним игроком** в комнате (и при необходимости **одна команда** с одним участником — на усмотрение реализации для smoke-тестов UI и логики поля).
- Остальная логика матча (тики, краска, Challenge, отрисовка поля) может работать в **упрощённом** сценарии; награды алмазами в solo-debug **рекомендуется отключать** или помечать как не выдаваемые, чтобы не плодить начисления из тестов.
- В UI обязательна **явная индикация** (баннер / watermark): **«DEBUG: solo lobby»** (или аналог), чтобы не перепутать с боевым режимом.

**Безопасность:** в продакшене значение флага **false** по умолчанию; при `true` в prod-окружении — **fatal** при старте приложения или игнор с логом ошибки (одна политика на проект).

**Чеклист перед продом:** `TEAM_TERRITORY_DEBUG_SOLO_LOBBY=0`; убедиться, что обходы п. 3.1 неактивны в prod-сборке и в CI; снять DEBUG-баннер из UI.

### 3.2 Поздний вход: матч уже идёт

Игрок подключился к комнате/каналу, когда фаза уже **«Идёт матч»** (старт по п. 3 п. 4 уже произошёл).

**Рекомендуемое поведение v1:**

1. **Наблюдатель:** игрок **не** добавляется в активный состав текущего матча (не тратит краску, не голосует за клетки). Он получает **тот же поток состояния**, что и участники: сетка **G×G**, таймеры, счёт по командам, при желании — состав игроков (без приватных данных).
2. **Очередь на следующий матч:** клиент показывает явный статус — например **«Идёт матч — вы в очереди на следующий»** и при необходимости **позицию в очереди**. После завершения текущего матча (любой исход по п. 8) сервер формирует **новый** лобби-раунд из очереди наблюдателей + оставшихся игроков по правилам п. 3 (или все желающие снова нажимают «Готов» — **одна политика на реализацию**).
3. **Альтернатива:** отдельная комната «следующий матч» — только если продуктово нужно изолировать потоки.

**Анти-абуз:** наблюдатель **не** влияет на таймеры и RNG текущего матча.

## 4. Поле и клетки

- Поле — квадратная сетка **G×G**, **`CELL_TOTAL` = G²**. Целое **G ≥ 10** задаётся **один раз** при старте матча по суммарному числу игроков (п. **4.1**). Клиент и API всегда передают актуальное **`G`**.
- Индексация клеток: **0 … CELL_TOTAL−1** либо **`(row, col)`**, `0 ≤ row, col < G`.
- Каждая клетка в матче находится в одном из состояний:
  - **Нейтральная** — ещё ни одна команда не закрепила закраску;
  - **Закреплённая за командой T** — учитывается в счёте команды T.

### 4.1 Размер поля от суммарного числа игроков

**Вход:** `P` — число игроков в матче **на момент старта** (все, кто вошёл в матч по правилам лобби п. 3). Размер поля **до конца матча не меняется**.

**Идея:** при **`P < 6`** поле остаётся **компактным 10×10**; при **6–10** игроках — **12×12**; дальше размер шагом **+2 к G** на каждый следующий диапазон, чтобы площадь росла с числом участников и матч не «сжимался» в слишком быстрое заполнение.

| Условие на `P` | **G×G** | `CELL_TOTAL` |
|----------------|---------|----------------|
| **P < 6** | **10×10** | 100 |
| **6 ≤ P ≤ 10** | **12×12** | 144 |
| **11 ≤ P ≤ 16** | **14×14** | 196 |
| **17 ≤ P ≤ 24** | **16×16** | 256 |
| **25 ≤ P ≤ 32** | **18×18** | 324 |

- Верхнюю границу **`P`** и/или **`G`** задать в конфиге матчмейкинга; при большем спросе — **очередь** или **несколько комнат**, а не бесконечный рост сетки.
- **Debug solo** (п. 3.1): при **`P = 1`** действует строка **`P < 6`** → **10×10**.
- **Баланс (опционально v2):** увеличивать `MATCH_MAX_SEC` и/или `PAINT_MAX` пропорционально **`CELL_TOTAL / 100`** после плейтестов; в v1 допустимо оставить значения из п. 12 без масштабирования.

**Связь с п. 6:** потолок закрасок за тик **`C`** рекомендуется масштабировать с площадью, например умножить базовое значение на **`max(1, CELL_TOTAL / 100)`** (коэффициент и округление — в конфиге), чтобы при **144+** клетках не замедлять игру чрезмерно.

**Версия v1 (рекомендуется для простоты и предсказуемости):** нейтральная клетка после первого успешного закрепления **больше не меняет команду** (нет перекрашивания чужого). Это снижает спорные ситуации и упрощает подсчёт.

### 4.2 Видимость поля (все команды)

- **Полная карта закрасок:** каждый игрок **всегда** видит на сетке **все** уже закреплённые клетки **всех** команд — **своими цветами** (красный / синий / зелёный / фиолетовый по правилам п. 2), включая клетки **соперников**. Нейтральные клетки — отдельным нейтральным стилем.
- Цель — понятная **тактическая картина**: наглядно видно, какая команда сколько территории заняла и насколько соперник(и) впереди или позади по площади, без «тумана войны» на чужие закраски.

*Расширение v2 (не обязательно в первой версии):* частичная видимость карты; перекрашивание чужих клеток с множителем стоимости и лимитами — отдельные ТЗ.

## 5. Ресурс «краска» (личный у игрока)

- У каждого игрока есть **запас краски** — целое число **от 0 до `PAINT_MAX`** (по умолчанию **10**).
- Закрепление **одной** клетки за свою команду стоит **1 единицу краски** (если закрепление прошло успешно по правилам п. 6–7).
- Пока краска **0**, игрок **не может** инициировать новое закрепление (кнопки/клики блокируются или уходят в очередь — см. ниже).

### 5.1 Пополнение краски

| Способ | Описание | Ограничения |
|--------|-----------|-------------|
| **Авторегенерация** | +1 краска каждые `REGEN_SEC` секунд, пока запас &lt; `PAINT_MAX` | Только в активном матче; на паузе матча — на усмотрение (обычно стоп) |
| **Покупка за алмазы** | Мгновенно +`BUNDLE` краски за цену `DIAMOND_COST` | Не выше `PAINT_MAX`; лимит **покупок за матч** (напр. **2**) чтобы не Pay-to-Win |
| **Challenge** | Мини-игра на время: **3 режима** (математика / правописание / кружки 1–10), см. п. 5.2 | Кулдаун между запусками; только в активном матче; начисление не выше `PAINT_MAX` |
| **Награда за действие** (опционально) | Например +1 краска за «первую закраску в раунде» — только если нужно ускорить скучные матчи | Конфиг выкл. по умолчанию |

Константы (`PAINT_MAX`, `REGEN_SEC`, `BUNDLE`, `DIAMOND_COST`, лимит покупок, параметры Challenge, `HUD_INK_POLL_SEC`) — **вынести в конфиг** сервера/режима.

### 5.2 Механика «Challenge» (пополнение краски через мини-игру)

Игрок вручную запускает режим кнопкой **«Challenge»**. Это **не** трата алмазов; награда — **+1 краска** за каждый засчитанный **ответ** (режимы 1–2) **или** за каждый **выигранный раунд** (режим 3), с потолком `PAINT_MAX`.

Challenge бывает в **трёх различных режимах**; при **каждом новом запуске** Challenge сервер **случайно** выбирает **ровно один** режим на всю сессию из **пяти** шагов — **режим 1 (математика)**, **режим 2 (правописание)** или **режим 3 (последовательность кружков 1–10)**. Все **5** шагов в сессии одного типа. Веса вероятностей (например 1/3 / 1/3 / 1/3) — в конфиге (`CHALLENGE_MODE_WEIGHT_*`).

#### 5.2.1 Параметры одного Challenge

| Параметр | Значение (дефолт) | Примечание |
|----------|-------------------|------------|
| Число шагов за один Challenge | **5** | Подряд в одной «сессии» (5 вопросов в режимах 1–2 **или** 5 раундов в режиме 3) |
| Время на **один** шаг | **5 с** | Режимы 1–2: одно задание за 5 с. **Режим 3:** за 5 с нужно завершить **весь** раунд (все 10 нажатий по порядку) |
| Кулдаун между **запусками** Challenge | **120 с** (2 мин) | Рекомендация: от **старта** предыдущего Challenge |
| Режим сессии | **Случайно** один из трёх | Один режим на все 5 шагов |
| Пауза между раундами | **3 с** (`CHALLENGE_ROUND_GAP_SEC`) | См. блок ниже — **обязательно** для режимов 1–3 |

**Переходы между раундами (все режимы 1–3).** Раунды **не должны** сменяться мгновенно: после **каждого** из пяти шагов **1…4** (после ответа на задание в режимах 1–2 или после завершения раунда кружков в режиме 3) клиент **обязан** показывать фазу **«итог раунда»**:

- **Панель итога:** явно **успех** или **неуспех** (иконка, цвет, короткий текст; при успехе — напоминание «+1 краска», при неуспехе — «без краски» / «ошибка» / «время вышло» в зависимости от причины).
- Затем **отсчёт 3 секунды** до следующего раунда (**3 → 2 → 1** крупным шрифтом или один общий прогресс-бар на 3 с).
- Только после отсчёта — **плавное появление** следующего задания (fade / slide / лёгкое масштабирование), без резкой подмены контента «в тот же кадр».

После **5-го** раунда вместо отсчёта к «раунду 6» показывается **финальная сводка** всей сессии Challenge.

**Сервер:** не засчитывать ответы / нажатия по **следующему** раунду (`k+1`) **раньше** момента `next_round_not_before`, выставленного после обработки раунда `k` (текущее время + `CHALLENGE_ROUND_GAP_SEC`), либо эквивалентная проверка по меткам времени в протоколе — чтобы нельзя было «проскочить» паузу читерским клиентом.

Пока активен Challenge, **рекомендуется** на клиенте не давать выбирать клетку поля (оверлей на весь экран мини-игры), чтобы не смешивать ввод ответа и закраску. На сервере закраска по тикам матча может оставаться разрешённой технически — по продукту лучше **игнорировать** выбор клетки на тики, пересекающиеся с открытым окном Challenge, или показывать предупреждение (на усмотрение реализации).

#### 5.2.2 Режим 1 — математика

- **Случайные примеры** с **целым неотрицательным ответом** (например `32 + 96 = ?`). Диапазон операндов и операции (`+`, `-`, `×` в простом диапазоне и т.д.) — **в конфиге**.
- Сервер **генерирует и проверяет** ответы; клиент **не** доверяет локальной генерации без сервера.

#### 5.2.3 Режим 2 — правописание (английский)

- Вместо примера показывается **английское слово** из утверждённого списка с **одной пропущенной буквой** (например `W_RD` вместо `WORD` или визуально слот `□` в нужной позиции).
- Игрок за **5 с** должен ввести **одну** недостающую **латинскую букву** (a–z). Сравнение с эталоном — **без учёта регистра** (`a` и `A` эквивалентны), если в конфиге не задано иное.
- Слова **только из фиксированного словаря** (см. п. 5.2.7); сервер подставляет пропуск в случайной допустимой позиции (не первый/не последний символ — **рекомендация**, чтобы избежать тривиальных подсказок; окончательное правило в конфиге).

#### 5.2.4 Режим 3 — последовательность (кружки 1…10)

- На экране отображаются **10 кружков** с цифрами **от 1 до 10** (каждая цифра ровно один раз). **Расположение** кружков на плоскости — **случайная перестановка** каждый раунд (чтобы не запоминать координаты); сервер передаёт клиенту конфигурацию раунда (идентификаторы кружков и подписи 1…10 либо координаты + метки).
- За **5 с** игрок должен **нажать (тапнуть) кружки в порядке 1 → 2 → … → 10**. Любая ошибка порядка **или** истечение времени до завершения цепочки — **раунд проигран** (**+0 краски** за этот раунд).
- Таких **раундов подряд — 5** за одну сессию Challenge. За **каждый выигранный** раунд (цепочка 1…10 завершена вовремя) — **+1 краска** (не выше `PAINT_MAX` после каждого начисления).
- Сервер **валидирует** последовательность нажатий (например события `circle_id` или метка цифры + время относительно старта раунда); клиент **не** определяет исход раунда локально без подтверждения сервера.

#### 5.2.5 Поток (сервер — источник истины)

1. Игрок нажимает **Challenge**. Сервер проверяет: матч активен, кулдаун истёк, нет уже идущего Challenge у этого игрока.
2. Сервер выбирает **режим** сессии (**1** / **2** / **3**) по вероятностям из конфига.
3. **Режимы 1 и 2:** сервер создаёт сессию с **идентификатором**, **типом режима**, **номером задания 1…5** и ожидаемыми ответами; клиенту отдаёт первое задание и дедлайн. После ответа (или таймаута) сервер возвращает результат раунда и **`next_round_not_before`**; клиент показывает **панель итога** и **отсчёт 3 с** (п. 5.2.1), затем выдаётся задание `k+1` с новым дедлайном **5 с**. За верный ответ — **+1 краска**.
4. **Режим 3:** для раунда `k` = 1…5 сервер выдаёт раскладку кружков и дедлайн **5 с**. После исхода раунда — то же: результат, **`next_round_not_before`**, на клиенте панель итога + **3 с**, затем следующий раунд с новой перестановкой. Начисление **0 или +1 краска** за раунд.
5. После **5-го** шага сначала **итог 5-го раунда** и при необходимости короткая анимация, затем **финальная сводка** Challenge; сессия **закрывается**; кнопка снова доступна после **кулдауна 2 мин**.

**Итого за один Challenge:** до **+5 краски** при идеале во всех режимах.

#### 5.2.6 Клиент: UX и анимации

- Кнопка **Challenge** неактивна на кулдауне: **таймер** «доступно через …».
- Краткая подсказка при старте сессии: **«Режим: примеры»**, **«Режим: слова»** или **«Режим: порядок»** (или иконки).
- **Обязательно для всех режимов:** между раундами **1→2**, **2→3**, …, **4→5** — **не мгновенная** смена экрана: сначала **панель итога** раунда (успех / неуспех), затем **отсчёт 3 с**, затем **плавное** появление следующего задания (см. п. 5.2.1).
- **Режим математика:** карточка с выражением, поле ввода **числа** / цифровая клавиатура; между карточками примеров — та же последовательность «итог → 3 с → следующий пример».
- **Режим правописание:** карточка со словом и визуально выделенным пропуском, ввод **одной буквы** (латиница); между словами — так же «итог → 3 с → следующее слово».
- **Режим последовательность:** 10 кружков с цифрами, **визуально разнесённые**; при верном тапе — лёгкий **pulse** на кружке; при ошибке — **shake** всей группы и мгновенный проигрыш раунда; между раундами — **итог → 3 с**, затем **красивый переход** к новой раскладке кружков (fade, slide, лёгкое масштабирование); **индикатор 5 с** на активный раунд (один прогресс на 10 нажатий).
- Во время активного ввода (окно **5 с**): индикатор времени на текущий раунд; по желанию — particle при верном ответе (режимы 1–2) или при завершении цепочки (режим 3).
- После пятого раунда — **финальная сводка** с сервера (сколько верно / сколько раундов выиграно, сколько краски).

#### 5.2.7 Контент: словарь для режима правописания

- Для режима 2 требуется **набор из не менее 1000 английских слов** (леммы/обычные слова), единый для продакшена: файл или таблица в репозитории / CI, **без дубликатов**, с фильтрацией по длине (например **длина ≥ 4** символов, только `[a-z]` после нормализации).
- Исключить по возможности: оскорбления, бренды, имена собственные (ручной или полуавтоматический просмотр выборки).
- Лицензия источника слов (например список частотных слов с открытой лицензией) — **зафиксировать** в `README` или комментарии к ассету.
- *Расширение:* второй пул на 1000+ слов для смены сезона — отдельная задача.

#### 5.2.8 Анти-абуз и честность

- Повторный старт Challenge до истечения кулдауна — **отказ** с кодом ошибки.
- Ответы / нажатия принимаются **только в окне**, привязанном к `challenge_id`, индексу шага (**или** номеру раунда в режиме 3) и **режиму**; клиент не может подменить тип режима после старта сессии.
- В режиме 3: нельзя засчитать нажатие **до** старта раунда или **после** дедлайна; порядок меток **строго 1…10** на сервере.
- Не принимать действия по раунду `k+1`, пока не истёк интервал **`CHALLENGE_ROUND_GAP_SEC`** после результата раунда `k` (см. п. 5.2.1).
- Опционально: лимит **Challenge за матч** (например 6) — конфиг.

### 5.3 Панель «мои краски» и сводка по краске соперников

- В UI на **той же панели**, где показан **личный** остаток краски игрока, **под** этим значением отображается **динамический** показатель **суммарного запаса краски у соперников**:
  - **Минимум (v1):** одно число — **сумма** текущих остатков краски по **всем игрокам, не входящим в мою команду** (все «вражеские» команды вместе).
  - **Расширение UX:** при **3–4 командах** дополнительно или вместо одной суммы — отдельная строка на **каждую вражескую** команду (например «Синие (всего): 12»), чтобы видеть вклад каждой фракции.
- Данные должны **обновляться регулярно**, чтобы игрок **примерно каждые 2–3 секунды** видел актуальные цифры: интервал **`HUD_INK_POLL_SEC`** (дефолт **2.5 с**, в конфиге ориентир **2…3 с**) — таймер опроса или **throttle** на потоке снапшотов WebSocket. Допускается **дополнительное** обновление показателя при каждом приходе состояния матча (тик), если это не нарушает throttle по нагрузке на клиент.
- При изменении числа относительно предыдущего показа — лёгкая **визуальная подсветка** (краткий pulse / смена оттенка цифры), чтобы динамика была заметна.
- Сервер является **источником истины** для сумм; клиент не суммирует по чужим игрокам из кэша без сверки с сервером.

## 6. Анти-бurst: матч не должен заканчиваться «в один миг»

Проблема: при большом числе игроков суммарно доступно много краски; если все тратят её одновременно, все **`CELL_TOTAL`** клеток могут заполниться за короткий интервал.

**Обязательные правила v1:**

1. **Тики матча**  
   Матч делится на **дискретные тики** длительностью `TICK_MS` (например **4000–8000 мс**). Все успешные закрепления клеток применяются **только на границе тика** сервером.

2. **Не более одной попытки закрепления с игрока за тик**  
   За один тик игрок может **выбрать одну** целевую клетку (или «пропуск»). Если UI отправил несколько целей — сервер принимает **последнюю валидную** до дедлайна тика или первую — **одна политика, зафиксированная в API**.

3. **Глобальный лимит закреплений за тик**  
   Пусть `C = min(GLOBAL_CAP, f(число_живых_игроков))`, где например  
   `GLOBAL_CAP = 12`,  
   `f(n) = 4 + floor(n / 3)` — чтобы при росте игроков поле не заполнялось слишком быстро. Параметр **`GLOBAL_CAP`** (и при необходимости итоговое **`C`**) масштабировать с **`CELL_TOTAL`** по правилу из **п. 4.1**.  
   Если в тик поступило больше валидных заявок, чем `C`, сервер отбирает заявки:
   - **по одной на игрок** уже выполнено в п.2;
   - среди оставшихся — **случайная перетасовка с весом** (например, у кого меньше закреплённых лично клеток в матче — чуть выше шанс) **или** строго **порядок по времени** прихода заявки на сервер.  
   Не прошедшие заявки: **краска не списывается**, игроку показывается причина «тик переполнен, попробуйте в следующем».

4. **Нейтральная клетка только одна команда**  
   Первая успешная заявка на клетку в тике выигрывает конфликт; остальные на ту же клетку в том же тике — отказ, краска не тратится.

Итог: минимальное число тиков до заполнения поля **≥ ceil(CELL_TOTAL / C)**; при **`CELL_TOTAL` = 100** и **`C` ≤ 12** это **≥ 9 тиков**; при тике 6 с — ориентир **≥ 54 с** чистого времени для базового поля — плюс реген и лимиты краски удлиняют матч (на больших полях — пересчитать от фактических **`CELL_TOTAL`** и **`C`**).

## 7. Валидность хода

Закрепление в тике **разрешено**, если:

- матч в фазе «идёт»;
- игрок в составе матча и не отключён (политика AFK отдельно);
- у игрока **краска ≥ 1** на момент применения тика;
- клетка **нейтральная** (v1);
- клетка в пределах **0 … CELL_TOTAL−1** (или валидная пара `(row, col)` при текущем **`G`**);
- игрок не превысил лимит попыток за тик (п.6.2).

Иначе — отказ, краска не списывается (кроме случаев, где списание намерения отложено — лучше **не списывать** при отказе).

## 8. Окончание матча и подсчёт

**Условие завершения (любое из):**

1. **Все `CELL_TOTAL` клеток** закреплены за командами.
2. **Лимит времени матча** `MATCH_MAX_SEC` истёк — подсчёт по текущему полю.
3. **Техническое завершение** (осталось &lt; 2 активных команд из-за дисконнектов) — см. п. 10.
4. **Простой матча (stale):** см. **п. 8.1** — истекло **`MATCH_STALL_IDLE_SEC`** без «значимой активности» участников; матч завершается **ничьёй по простою**, подсчёт по текущему полю (без новых закрасок в момент срабатывания).

### 8.1 Простой матча и обратный отсчёт (ничья по бездействию)

**Цель:** не оставлять комнату в вечном матче, если участники перестали взаимодействовать с игрой (ушли AFK, вкладка свёрнута, ссора и т.д.).

**Участники:** только игроки в **активном составе текущего матча**. Действия **наблюдателей** (п. 3.2) **не** сбрасывают и **не** продлевают таймер простоя.

**Значимая активность** (сбрасывает таймер простоя на **0**):

- успешное или неуспешное **закрепление** по правилам тика (п. 6–7) — отправка заявки на клетку в окне тика;
- **Challenge:** старт сессии, отправка ответа / нажатие по режиму 3;
- **покупка краски за алмазы** в рамках матча.

*Опционально:* любое нажатие «Готов» к следующему тику — только если в протоколе есть явное действие игрока; иначе не считать, чтобы не продлевать матч «пустыми» кликами.

**Таймеры:**

- **`MATCH_STALL_IDLE_SEC`** (дефолт **300 с** = **5 мин**) — если с момента последней **значимой активности** прошло ≥ этого времени, матч **немедленно** завершается по п. 8 п. 4.
- **`MATCH_STALL_WARN_BEFORE_SEC`** (дефолт **60 с**) — за столько секунд **до** срабатывания простоя сервер переводит матч в фазу предупреждения и рассылает **`stall_deadline_at`**; клиент показывает **обратный отсчёт** (крупные цифры / кольцо), текст вроде: **«Нет действий — через N с ничья по простою»**. После появления предупреждения любая значимая активность **снимает** предупреждение и обнуляет таймер.

**Исход:** зафиксировать причину **`stale_idle`**; **ничья по счёту клеток** (или ничья варианта **C** из п. 8 — **0 алмазов** всем участникам матча, если не задано иное в конфиге). Наблюдатели в очереди **не** получают награду за этот матч.

**Подсчёт:**

- Для каждой команды: `score = число клеток её цвета`.
- Победитель: команда с **максимальным** `score`.
- **Ничья** (две и более команд с одинаковым максимумом):
  - вариант A: **дополнительный овертайм** — `OVERTIME_TICKS` тиков, нейтральные клетки если есть, иначе только смена правил в v2;
  - вариант B: **поделить награду** поровну между всеми игроками команд-победителей с максимальным счётом;
  - вариант C: ничья — **фиксированная меньшая награда** всем участникам.  

Рекомендация для v1: **вариант B или C** (проще в реализации, без нового поля).

## 9. Награды (алмазы)

- Каждому игроку **победившей команды** (после разрешения ничьей по п.8): **+5 алмазов** (конфиг).
- Проигравшие команды: **0** (или косметика/опыт — вне скоупа, если не оговорено).
- Завершение по **простою** (`stale_idle`, п. 8.1): **рекомендация v1** — **0 алмазов** всем **активным** участникам матча (отдельно от ничьей по равному счёту клеток); иначе зафиксировать явное правило в конфиге.
- Начисление **один раз** на матч, с **идемпотентностью** (повторный запрос не дублирует выдачу).
- **Анти-абуз:** минимальное время матча / минимум совершённых тиков / минимум закреплений с игрока для получения награды — флаг в конфиге (например награда только если игрок **участвовал минимум в 3 тиках**).

## 10. Отключения и AFK

- **Дисконнект:** игрок помечается офлайн; его краска не тратится без действий; команда продолжает матч.
- Если в команде **0 онлайн-игроков** и нет политики ботов — команда не может набирать очки новыми закреплениями (уже закреплённые клетки остаются).
- Если **все игроки одной команды** вышли — опционально **ранний проигрыш** этой команды не обязателен: поле может дойти до конца по таймеру.
- **AFK кик из матча** (опционально): не нажимал «готов» к действию N тиков подряд — исключение из активных заявок без штрафа краски.
- **Связь с п. 8.1:** игрок в активном составе, но **офлайн** — не генерирует значимую активность; таймер простоя **не** ставится на паузу из-за одного дисконнекта (матч может завершиться по простою, если остальные тоже молчат).

## 11. Состояние клиента (кратко)

Клиент отображает:

- режим **наблюдателя** и **очередь на следующий матч** (п. 3.2): баннер, позиция в очереди, отключённые действия закраски;
- при фазе предупреждения **простоя** (п. 8.1): **обратный отсчёт** до ничьей и краткое пояснение; у участников — без блокировки ввода, если тик ещё принимает заявки;
- сетку **G×G** (значение **`G`** с сервера): **все** закреплённые клетки **всех** команд **с их цветами** (см. п. 4.2), плюс нейтральные клетки; при **G > 12** допускаются **масштабирование** поля под экран и/или **панорамирование** (pinch-to-zoom), без потери кликабельности;
- свой запас краски и таймер до следующего тика / регенерации;
- **под** личным запасом краски — **суммарную краску соперников** (и при необходимости разбивку по вражеским командам), с обновлением **каждые ~2–3 с** и лёгкой анимацией при смене значения (п. 5.3);
- кнопку выбора клетки на текущий тик (или drag — не важно для ТЗ);
- состав команд и счёт по клеткам (можно обновлять раз в тик);
- магазин пополнения краски за алмазы (с лимитом за матч);
- кнопку **Challenge**, таймер кулдауна, полноэкранный/модальный режим **5 шагов по 5 с**; индикация текущего **режима** (математика / англ. слово / порядок 1–10); UI: число, одна буква или **10 кружков**; после **каждого** раунда 1…4 — **панель итога** и **отсчёт 3 с** до следующего (п. 5.2.1); анимации смены карточки/раскладки и таймера.

### 11.1 Визуальный язык, UX и анимации (обязательный уровень качества)

Интерфейс игры — **не утилитарная заготовка**, а полноценный продуктовый экран: **современный, лаконичный, понятный** новичку без инструкции, с **согласованной** эстетикой на всех экранах (лобби, матч, Challenge, победа/поражение).

**Визуальный язык**

- **Типографика:** 1–2 шрифта (основной + акцент для цифр/таймеров), крупные **читаемые** числа для таймеров и краски; иерархия заголовок → вторичный текст → подсказка.
- **Цвет:** палитра строится вокруг **цветов команд** (п. 2) + нейтральный фон и **семантика** (успех / ошибка / нейтраль); достаточный **контраст** текста и иконок (ориентир — доступность, без «серого на сером»).
- **Сетка и отступы:** единая **8-pt (или 4-pt) сетка** отступов; скругления и тени **в одном стиле** на карточках, панелях, кнопках; визуальная «глубина» через лёгкий elevation, без перегруза.
- **Иконки:** простые узнаваемые пиктограммы; состояния disabled / loading / active явно отличаются.

**Понятность UX**

- На экране **один главный фокус** в момент времени (например выбор клетки **или** Challenge — не конкурирующие равнозначные блоки).
- **Подсказки в контексте** (короткие строки под кнопкой, tooltip по долгому нажатию — по желанию): что делает тик, зачем краска, что значит счётчик соперников.
- **Пустые и загрузочные состояния:** skeleton / shimmer при подгрузке матча; если данных ещё нет — дружелюбный текст, а не пустой экран.
- **Touch:** цели нажатия **не меньше** рекомендуемого минимума (~44×44 px логических); размер **визуальной** клетки на экране подбирается так, чтобы при текущем **`G`** не было систематических промахов (см. масштаб/панораму для **G > 12**); для Challenge — запас вокруг кружков.

**Анимации и переходы**

- **Переходы экранов** (лобби → матч, открытие Challenge, итог раунда): **плавные** (ориентир **150–400 мс**, easing `ease-out` / мягкая кривая), без рывков и мгновенных «прыжков» макета.
- **Клетки поля:** появление нового цвета — короткая **заливка / scale-in / ripple** от точки действия; при наведении/выборе — лёгкий **highlight**; при смене тика — **микро-пульс** доступности хода (если применимо).
- **Числа и счётчики** (краска, соперники, таймер тика): при изменении — **краткий** count-up / pulse / цветовой flash **&lt; 0,5 с**, не отвлекающий от игры.
- **Кнопки:** press-state (scale 0.97–0.98), disabled — пониженная непрозрачность + `cursor`/aria.
- **Challenge** — см. п. 5.2.6 и 5.2.1; анимации там — **обязательная** часть качества, не опция.
- **Принцип:** движение **осмысленное** (подтверждение действия, направление внимания), без бесконечных loop-анимаций на фоне и без «цирка» на каждый клик.

**Обратная связь и завершение**

- Успех / ошибка / таймаут — **визуально и при желании звуко** (короткий sfx, выключаемый в настройках).
- Экран **победы / поражения / ничьей** — отдельная композиция: команда-цвет, итог по клеткам, награда алмазами, **одна** главная кнопка «Дальше» / «В хаб»; короткая **celebration**-анимация при победе (конфетти optional, умеренно).

**Техническая аккуратность**

- **prefers-reduced-motion:** при системной настройке «уменьшить движение» — сократить или отключить декоративные анимации, сохранив смену состояний.
- Стабильный **layout** — нет сдвига блоков при подгрузке цифр (фиксированные ширины для счётчиков или tabular figures).
